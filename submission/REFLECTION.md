## 5. Bonus - The single change that mattered most

> **Most important section.** Pick **one** change from the bonus track (build flag, thread sweep, quant pick, GPU offload, KV-cache quantization, speculative decoding, or any challenge in `BONUS-llama-cpp-optimization/CHALLENGES.md`) that produced the biggest speedup on your machine.

**Change:** Tuned the CPU decode thread count to the physical core count instead of oversubscribing all logical threads.

**Before vs after** (paste 2-3 lines from sweep output):

```
before: -t 24 -> 2.8 tok/s
after:  -t 6 -> 19.8 tok/s
speedup: ~7.1x
```

**Why it works** (1-2 short paragraphs - this is the part the grader reads carefully):

The decode path is memory-bandwidth bound on this CPU. Once the worker count reaches the physical core count, adding more threads mostly increases contention for the same cache and memory channels instead of creating more useful throughput.

That is why `-t 6` wins on this machine. `-t 24` spreads work across hyperthreads and oversubscribes the shared execution resources, so throughput falls instead of rising. The curve is the expected shape for a bandwidth-limited workload.

## 6. (Optional) Most surprising thing

The most useful surprise was how quickly the server hit a hard throughput ceiling once the native slots were full. More concurrency did not improve latency; it only increased queuing.
